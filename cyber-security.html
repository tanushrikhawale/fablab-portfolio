Networking — 7-day Module

Detailed theory, labs and applied examples (timeline follows Renuka's doc). :contentReference[oaicite:1]{index=1}

This page documents the seven-day networking module: each day builds on the previous one, moving from fundamental concepts to packet-level diagnostics and finally a small embedded-network project. The structure and topics follow the reference module. :contentReference[oaicite:2]{index=2}

Day 1 — Foundations: what a network is and how information travels

We started with the minimal model: a sender, a receiver, a message, a medium and a shared protocol. Practically, this means breaking down how bits (0/1) travel across copper, fibre or radio and are reassembled at the other end. The session emphasized the idea of *reliable transmission*: ordering, completeness and correctness are the goals.

Communication modes were explained with physical analogies: simplex (one-way like TV broadcasting), half-duplex (two-way but one direction at a time, like walkie-talkies) and full-duplex (two-way simultaneously, like modern VoIP). Understanding modes clarifies why some links can transmit both directions at once and others cannot.

We also contrasted analog vs digital transmission: analog is continuous (audio waveforms), whereas digital encodes information in discrete symbols — digital is less sensitive to noise, easier to encrypt, and simpler to multiplex in modern networks. Practical characteristics we measured were throughput (speed), latency (delay), reliability and error rate.

Activity: sketched a home network: router, phone, laptop. Traced a sample WhatsApp packet across local Wi-Fi → ISP → recipient; this clarified real-world packet traversal. :contentReference[oaicite:3]{index=3}

Day 2 — Types of networks & nature of data

Day 2 zoomed out. We classified networks by scope and function: PAN (personal area networks like Bluetooth), LAN (home/office), MAN (city-wide) and WAN (ISP/internet). The key differences are scale, management and QoS (quality-of-service) requirements.

We compared architectural models: client-server (centralized services, e.g., web servers) vs. peer-to-peer (distributed sharing, e.g., some file-sharing systems). The choice affects latency, resilience and how state is managed.

We also discussed the *nature of data* that flows on networks: structured transactional data (databases), observed sensor data (slow but frequent IoT telemetry), and inferred or analytic data (batch processed). Understanding data kinds helps select transport, storage and security strategies.

Use-case note: low-latency use cases (financial trading) require different links and tuning than bulk file transfer — that distinction guided our lab choices. :contentReference[oaicite:4]{index=4}

Day 3 — OSI model demystified (the 7 layers)

Day 3 was the OSI deep-dive. The OSI model separates networking into seven layers so we can reason about responsibilities: Physical (bits & media), Data Link (MAC frames, switching), Network (IP routing), Transport (TCP/UDP reliability), Session/Presentation (connection & encoding) and Application (HTTP, FTP, DNS).

We studied representative protocols for each layer: Ethernet at Data Link; IP at Network; TCP and UDP at Transport; and HTTP/HTTPS at Application. A core exercise was *encapsulation*: an application payload gets wrapped with headers (transport → network → link) as it goes down the stack and then unwrapped by the receiver.

For hands-on, we traced a packet using ping and traceroute, observing how TTL and hops appear. This helped build intuition for routing behavior and which layer is responsible for errors or congestion.

Commands:

# basic diagnostics
ping 8.8.8.8
traceroute example.com

Day 4 — Inside the web: DNS, HTTP & REST

The focus moved to how the web actually works: DNS resolves human-friendly names to IP addresses — it's the internet's lookup service. We discussed DNS caching, TTL, authoritative vs recursive resolvers, and how caching speeds up repeated lookups.

HTTP and HTTPS were explained at the request/response level: methods (GET, POST, PUT, DELETE), headers (Content-Type, Authorization) and status codes (2xx success, 4xx client error, 5xx server error). We also covered RESTful API design patterns and idempotency concerns.

Lab tools included nslookup / dig to inspect DNS records and browser developer tools (Network tab) to watch HTTP handshakes and headers.

Example (quick lookups):

nslookup example.com
curl -I https://example.com

Practical exercise: observed how caches and CDN edge nodes affect response times and how to read response headers for caching policies. :contentReference[oaicite:6]{index=6}

Day 5 — DHCP, static vs dynamic addressing & review

DHCP (Dynamic Host Configuration Protocol) automates IP assignment. We learned the handshake phases: Discover → Offer → Request → Acknowledge. DHCP also provides gateway, DNS servers and lease time which controls how long an address is valid.

The class contrasted static addressing (useful for servers & infrastructure) with DHCP (convenient for end devices). We inspected a router’s DHCP lease table to see addresses, MACs and expiry times.

Lab tip: when diagnosing client connectivity, check the lease, gateway and DNS assigned by DHCP before assuming application-level faults. :contentReference[oaicite:7]{index=7}

Day 6 — Low-level protocols (ARP, ICMP) and diagnostics

We dug into ARP (Address Resolution Protocol) — how an IP address is translated to a MAC address on a LAN. ARP uses broadcast queries and caches the reply in an ARP table. That explains why devices on the same LAN can send frames directly using MAC addresses.

ICMP (Internet Control Message Protocol) supports diagnostics such as echo requests/replies (used by ping) and TTL-exceeded messages (used by traceroute). Using Wireshark, we inspected ARP requests and ICMP echo packets to see the actual bytes on the wire.

Useful commands:

arp -a        # show local ARP table
ping 192.168.1.1
sudo wireshark  # capture & inspect packets

Activity: identified a misconfigured gateway by observing ARP replies and missing ICMP responses. :contentReference[oaicite:8]{index=8}

Day 7 — Project: embedded networking (Zynq FPGA example)

On the final day the class combined networking fundamentals with embedded systems: a Zynq FPGA-based demonstrator that implements minimal TCP/IP stack components. The project required putting DHCP, ARP and basic ICMP handling together so the device could boot, get an address and respond to pings.

This exercise bridged software and hardware: we configured the programmable logic for network I/O and ran a lightweight network stack on the processing system. The outcome reinforced that the same protocols used on servers are essential on embedded & IoT devices — with extra attention to resource constraints and boot-time networking.

Project takeaway: applying DHCP + ARP + ICMP in constrained hardware clarified timing, retries and how to debug using simple packet captures. :contentReference[oaicite:9]{index=9}

✅ Learning outcomes

  • Clear mental model of the OSI layers and which layer handles what responsibilities.
  • Comfort with IP addressing, subnetting and DHCP for practical network design.
  • Hands-on experience with Packet Tracer, Wireshark and command-line diagnostics (ping/traceroute/arp).
  • Awareness of security & operational trade-offs when deploying embedded networked devices.
