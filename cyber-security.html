Networking – 7-Day Module

This module was a structured, 7-day introduction to Networking fundamentals. Each day focused on a specific topic, starting with basics and culminating in a working embedded project. Through a mix of theory, Packet Tracer labs, and command-line practice, I learned how devices communicate, how protocols operate, and how to analyze traffic.

Day 1 – Introduction to Networking

Networking is about enabling communication between devices. On the first day, I learned about the essential components of any communication system — sender, receiver, medium, message, and protocol. We discussed why protocols are needed to standardize communication across different devices and manufacturers.

I also studied transmission modes: simplex (one way), half-duplex (both ways but not at the same time), and full-duplex (simultaneous two-way). We related these to real-world examples like radios, walkie-talkies, and phone calls.

Activity: Drew a network showing how my laptop, smartphone, and printer communicate via a Wi-Fi router.

Day 2 – Types of Networks & Data

On the second day, we classified networks by scale: PAN (Bluetooth devices), LAN (home/office networks), MAN (city-wide), and WAN (global Internet). We also distinguished between client-server models, where central servers provide services, and peer-to-peer networks, where devices share directly.

Another key discussion was on the nature of data: structured data like banking transactions, unstructured data like social media, and real-time data from IoT devices. Understanding data types helped me connect networking to real-world applications.

Activity: Identified and presented 3 use-cases of LAN and WAN in daily life.

Day 3 – OSI Model

The OSI model describes networking in 7 layers: Physical, Data Link, Network, Transport, Session, Presentation, and Application. I learned how each layer has its own responsibility — for example, the Transport layer (TCP/UDP) ensures reliable delivery, while the Network layer (IP) handles addressing and routing.

Encapsulation was a core takeaway: when data moves down the OSI stack, headers are added, and when it reaches the receiver, those headers are stripped layer by layer. This logical division makes complex communication easier to manage and troubleshoot. 

# Checking connectivity
ping 8.8.8.8
traceroute www.google.com

Activity: Used ping and traceroute to understand packet travel and delays.

Day 4 – HTTP & DNS

Day 4 focused on application-layer protocols. I explored DNS, which translates domain names into IP addresses, and understood the difference between recursive and authoritative DNS servers. TTL values explained how long results are cached.

We also studied HTTP/HTTPS, request/response cycles, and status codes (200, 404, 500). With developer tools, I could see live headers and responses, making the concepts practical. 

# DNS Lookup
nslookup openai.com

# View HTTP headers
curl -I https://example.com

Activity: Resolved domain names and inspected HTTP responses for caching info.

Day 5 – DHCP

DHCP automates IP assignment. We studied the DORA process: Discover, Offer, Request, Acknowledge. This process reduces manual configuration and prevents conflicts in networks with many devices. We compared static IPs vs dynamic IPs and discussed where each is useful. 

# Example DHCP lease
IP Address : 192.168.1.25
MAC Addr   : 00:1A:2B:3C:4D:5E
Lease Time : 24 hours

Activity: Observed DHCP leases on a local router and manually configured a static IP for comparison.

Day 6 – ARP & ICMP

The sixth day introduced ARP (Address Resolution Protocol), which maps IP addresses to MAC addresses inside a LAN. We also revisited ICMP (used in ping) for diagnostics. This helped me understand how devices locate each other physically within a network. 

# View ARP table
arp -a

# Test reachability
ping 192.168.1.1

# Capture packets
sudo wireshark

Activity: Inspected ARP cache entries and saw real-time ICMP packet exchanges in Wireshark.

Day 7 – Embedded Networking Project

On the final day, we applied everything in an embedded environment using a Zynq FPGA board. The board was programmed to acquire an IP via DHCP, resolve ARP requests, and respond to ICMP pings. Seeing the protocols I had studied come alive in hardware was a powerful way to connect theory to practice.

Outcome: Demonstrated a working embedded system communicating over the network, showcasing complete understanding of the week’s lessons.

  • Configured and tested DNS, HTTP, and DHCP concepts.
  • Practiced ARP and ICMP with real diagnostic tools.
